**Continued same session — Critic findings, Phase 3 scheduler, component design draft:**

- **Phase 3 scheduler shipped** (10 weeks budgeted; landed in this turn):
  - **`batch<T>(fn): T`** — group writes; watches see the final state. Nests; only outermost flushes. Reads inside the batch see writes performed earlier in the batch. Mid-batch throw flushes the partial state (matches Solid's behavior; atomic semantics deferred to Phase 5 transitions).
  - **`flush(): void`** — synchronously drain both queues. No-op inside a batch.
  - **`watch(fn, { defer: true })`** — opt-in microtask coalescing. Multiple writes between microtask boundaries collapse into one watch run. The default-stability of `defer: false` is now committed.
  - **`peek<T>(s)`** — untracked read. Doesn't subscribe the current observer.
  - Internal refactor: replaced the single `scheduledWatches` queue with two queues (`syncQueue`, `deferredQueue`) plus `batchDepth` and `microtaskScheduled` flags. `propagateMark` routes scheduled watches by their `deferred` flag.
  - 23 new scheduler tests (`tests/unit/scheduler.test.ts`).
  - Updated test plan with 14 Phase 3 invariants (3.1-3.14) reflecting the shipped design (sync-default + opt-in defer, not the original "deferred-default with phase semantics" plan).

- **Critic finding caught a test design error:** I had a test for "watch that writes to itself triggers the round limit." It didn't — the COMPUTING-state short-circuit prevents a watch from rescheduling itself while it's running. Real infinite loops only happen via *cross-watch mutual feedback* (watch A writes B's source, watch B writes A's source). Updated the test scenario; both tests now correctly trigger the limit.

**Persistence v0.1 → v0.2.**
- v0.1: `PersistenceAdapter<T>` contract (`load` + `save`), `persistedState(adapter)` wrapping a `state` with a `watch` that auto-saves, `localStorageAdapter` and `memoryAdapter`. 15 tests.
- v0.2: optional `observe(callback): Disposer` on the adapter contract. `persistedState` subscribes if present and re-loads on external change. Cycle break via closure-local `applyingRemote` flag — without it, A's save → broadcast → B's reload → B's save → broadcast loops forever. **`crossTabAdapter(inner, channelName)`** wraps any adapter with `BroadcastChannel` sync. Per-spec the sender doesn't receive its own broadcasts, so the suppression flag covers the receiver and the cycle is fully closed. 23 tests including end-to-end with two `persistedState`s on the same channel asserting non-loop semantics.
- Commonplace integrated: `persistedNoteStore` now wraps `localStorageAdapter` with `crossTabAdapter`. Open two tabs of the book; edits in one tick into the other within ~1 frame. Conflict policy is last-write-wins for v0.2; CRDT/OT lives in the deferred sync-server adapter.

**Routing system shipped, then deepened.**
- v0.1: `Router` interface (`path/navigate/replace/back/forward`), `hashRouter()` with `hashchange` listener, `memoryRouter()` for tests/SSR, `RouterCap` capability. 17 tests. Hash mode chosen over history because it works on any static host without server config.
- v0.2: reactive `segments()` and `query()` accessors, cached via derived state — multiple readers share one parse per path-change. Defensive clone of `URLSearchParams` per `query()` call so caller mutations don't leak. `parsePath(path)` free utility for off-router parsing. **`updateQuery(changes, options?)`** merges changes into the current query while preserving the path; `null` deletes a key. 35 tests total.
- Commonplace: routes simplified from `/notes/<id>` to `/<id>` (no other resource type, prefix was overhead). Tag filtering wires through routing's query: tag chips in the sidebar are real `<button>`s that call `router.updateQuery({ tag })`. Active-filter chip in the header (× to clear). Stale-URL guard via a `watch` that redirects if `selectedId()` points at a missing note. `onSelect` preserves the active query so filter sticks across selections.

**Search v0.1.** `searchable(items, { fields, caseSensitive? })` — substring, case-insensitive, AND-of-tokens. ~30 LOC including comments. The interesting move was *removing* search from `NoteStore`: storage is storage, retrieval is retrieval. `NoteStore` lost the `search` method; `searchNotes(store)` is a 3-line helper composed at the call site using `@place/search`. 11 tests.

**Reactivity Phase 5 partial — `resource(loader)` async-as-pending.** ~50 LOC, unblocked by no other work. **DX helpers — `wire`, `onKey`, `globalKey`.** All three earned their keep by appearing 2+ times in user code as boilerplate, then collapsing to a single call. None invented for a hypothetical future need.
- **`wire(state)`** / `wire(get, set)` — text-input two-way binding. `<input {...wire(query)} />` replaces 2 lines of `value={() => …}` + `onInput={(e) => … as HTMLInputElement …}`.

Auto-disposes via `onCleanup` when called inside a component body; returns the disposer for non-component contexts. Used in Sidebar (⌘K, ⌘N, ↑/↓) and App (Esc closes editor).

## 2026-05-04 — SSR layer, page-as-data, opinionated-but-explicit server, typed metadata + CSP, first-class Tailwind v4

The arc this session: the component system grew from "render layer only" into "render layer + SSR layer + opinionated-but-explicit server primitives." The whole sync-server demo's SSR path collapsed from ~80 lines of glue per page (manual handler + htmlShell + serverRouter + Bun.build + /client.js handler + querySelector hack to extract `now` from SSR'd DOM) down to a 50-line shared `home.page.tsx` plus literal one-line route registration `'GET /ssr/demo': homePage`.

**Direction of the session.** It started with "can this whole thing be server-first like Next SSR but better and more secure with the least to no attack surface?" and ended with the page-as-data + serve/boot triad shipping behind 232 component tests (554 across the platform). Every step pushed against a Next/Nuxt failure mode: file-system routing, `'use client'` magic, RSC complexity, built-in caches that span auth contexts, default-`'unsafe-inline'` CSP. Ours doesn't ship any of those.

**What landed (in order).**

**1. SSR foundation.** Five phases per [tender-booping-waffle plan](#) — Bun JSX runtime config (`bunfig.toml`), `handler(routeFn, options)` HTTP wrapper, build-static script, string-emitter pipeline (every built-in `View` got a `toHtml()` method so SSR runs in pure Bun without happy-dom), `renderToStream`, `hydrate(view, root)`. JSX SSR demo at `/ssr/demo` end-to-end. 60 SSR-related tests across `render-to-string`, `render-to-stream`, `handler`, `hydrate`, `ssr-edge-cases`, `html-shell`. The hydrate path adopts SSR'd DOM in place — element identity preserved, listeners + watches attached without recreation, `data-h` markers stripped post-hydration.

**2. Six edge-case bugs surfaced and fixed.** `keyed()` and `errorBoundary()` lacked `toHtml`/`hydrate`; nested element hydration was losing identity (now walks via slot when all children are hydratable Views — falls back to clear+remount only for mixed text/function children); old hydrate test asserted V0 clear-and-remount semantics. Each fix shipped with a failing-then-green test.

**3. DX helpers — `<Static>`, params threading, better hydrate error.** `Static` is Astro's islands inverted: default-hydrate, mark-static. The hydrate-mismatch error now lists the three most common causes by name (root-arg confusion, View divergence, post-SSR HTML mutation) — every user hits the root-arg one at least once. `Handler` type became `(req, params?) => Promise<Response>` so `:id`-style serverRouter captures flow into pages.

**4. The `page() / serve() / boot()` triad.** Single declarative object both server and client import: `page({ url, load, view, meta, styles, headers })`. `url(url, params)` runs on both sides (pure derivation); `load(ctx)` runs server-only and the result lands in a single inspectable `<script type="application/json" id="__place_load__">`; `view(props)` is the JSX. `serve()` bundles the client entry, dispatches Pages and raw handlers in one map, owns WebSocket/CORS via the `fetch` pre-router hook. `boot()` matches `location.pathname`, derives URL props the same way the server did, reads load data from the script tag, hydrates against `document.body`. The sync-server demo refactored: `client.tsx` collapsed to 15 lines, server.tsx SSR section collapsed from ~60 lines to 3.

**5. Typed metadata + styles.** Replaced the ad-hoc `shell: { title, css }` blob with typed `meta` (every key maps to one HTML element — title, description, og.*, twitter.*, icon, robots, canonical, theme-color, color-scheme, keywords, plus `extra: HeadEntry[]` escape hatch) and typed `styles` (`string` URL → `<link>`, `{ inline }` → `<style>`, or array). `meta` accepts a function `(props) => Meta` for dynamic titles per page (received the merged `{ ...urlProps, ...loadData }`).

**6. First-class Tailwind v4.** `serve({ tailwind: true })` lazy-imports the helper (apps that don't use it pay zero dependency cost), compiles via `@tailwindcss/node` + `@tailwindcss/oxide` once at startup, auto-injects the CSS into every page's `<head>`. The standalone helper `await tailwind({ content })` is sub-exported as `@place/component/tailwind`. Server-only Tailwind imports are kept out of the client bundle via `Bun.build({ external: ['@tailwindcss/node', '@tailwindcss/oxide', 'tailwindcss', 'lightningcss'] })` in `serve()`.

**7. Typed security headers + auto-Tailwind-hash CSP.** `serve({ security: 'strict' | 'standard' | 'none' | { ... } })` covers CSP (camelCase directives auto-kebab-cased, keywords auto-quoted), HSTS, Referrer-Policy, X-Content-Type-Options, X-Frame-Options, COOP/COEP/CORP, Permissions-Policy. The killer integration: when `tailwind` is enabled (default inline mode), the SHA-256 of the compiled CSS is auto-added to CSP `style-src` — strict CSP works without `'unsafe-inline'`. No other framework I know of does this auto-hash integration.

**What's deliberately not in scope.**

**Uncertain about:**

- Whether `boot()` should support nested routing (it currently looks at `location.pathname` once). Could iterate when a multi-route SPA demo lands.
- Whether the `extra: HeadEntry[]` escape hatch should be open or closed (currently a discriminated-union enum of `meta`/`link`/`script`/`style`). A user wanting `<base href>` would have to wait for it. Tradeoff is correctness (typed escape) vs flexibility (raw string).
- Whether a `tailwindVersion()` introspection method would be useful — Tailwind 5 will ship someday and apps may want to branch. For now, the abstraction is the answer.
