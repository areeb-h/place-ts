# Dev-cycle latency

Time from "edit a component file" to "next GET / reflects the change" via `bun --watch` server restart. place-ts has no module-level HMR; this is the practical edit→render dev cycle a developer experiences.

Generated by `bun scripts/bench-dev-cycle.ts`. The script synthesizes a 1000-component app, spawns the server under `bun --watch`, then performs the trials. Re-run to refresh the table — append-only so prior runs stay as a baseline.

## Methodology

- **Cold start:** spawn `bun --watch server.tsx` → first 200 OK on `GET /`. Includes module graph load, `Bun.build` of the client bundle, and `Bun.serve` startup. No Tailwind compile in the synthetic app.
- **Edit-bounce:** touch one component file with a unique marker → measure ms until `GET /` includes the marker. Repeated `--trials` times. p50/p95/p99 across the trial distribution.

place-ts ships no `import.meta.hot` plumbing. `bun --watch` restarts the process on file change. This is the supported dev cycle as of v0.5; module-level HMR with state preservation is roadmap 6.2.

## Comparison context

For frame of reference (from public bug reports / blog posts; not measured here):

- Next.js 14/15 on real apps: HMR routinely 25–45s per cycle ([discussion #58889](https://github.com/vercel/next.js/discussions/58889), [Northflank post-mortem](https://northflank.com/blog/why-we-ditched-next-js-and-never-looked-back), [discussion #85744](https://github.com/vercel/next.js/discussions/85744)).
- SvelteKit on large projects: LSP init ~1 minute ([kit#13455](https://github.com/sveltejs/kit/discussions/13455)).

Our claim should be made only after measuring. Don't market dev-cycle wins from this benchmark until at least 3 runs land in the table below and show a stable distribution.
